**每个游戏都内置美术替换系统，密码保护，实时预览。**

**不再让用户花时间找 BUG，所有验证 AI 自主完成。**

当用户提到以下关键词时自动触发：
- "做个游戏"
- "开发游戏"
- 具体游戏名称（如"雷霆战机"、"消消乐"）
- 任何游戏开发相关任务

## 🔄 完整工作流 (6 阶段)

```
1. 需求确认 → 2. 开发实现 → 3. QA 测试 → 4. 美术替换 → 5. 版本管理 → 6. 部署同步
```

---

## 📝 阶段 1：需求确认

### 必须执行
1. 创建 PRD 文档
2. 列出功能清单（P0/P1/P2 优先级）
3. 制定验收标准
4. **用户确认后才开工**

### PRD 模板
```markdown
# 游戏需求文档

**项目：** [游戏名称]
**版本：** V1.0.0

## 功能清单
| ID | 功能 | 优先级 | 验收标准 |
|----|------|--------|----------|
| F01 | ... | P0 | ... |

## 用户确认
- [ ] 功能清单正确
- [ ] 可以开始开发
```

### 🚨 禁止行为
- ❌ PRD 未创建就开工
- ❌ PRD 未经用户确认
- ❌ 擅自变更需求

---

## 🛠️ 阶段 2：开发实现

### 必须执行
1. 开发前检查清单
2. 执行开发（对照 PRD）
3. 中期汇报（50% 进度）

### 开发前检查清单
```markdown
- [x] PRD 已创建
- [x] PRD 已用户确认
- [x] 功能清单理解无误
- [x] 验收标准明确
- [x] 技术准备完成
```

### 中期汇报
**进度 50% 时主动汇报：**
- 已完成功能
- 进行中功能
- 遇到的问题
- 进度截图

### 🚨 禁止行为
- ❌ 不做开发前检查
- ❌ 进度 50% 时不汇报
- ❌ 遇到问题不请示

---

## 🧪 阶段 3：QA 测试（交叉测试）

### 三轮测试必须执行

#### 第一轮：test-master
- 单元测试
- BUG 扫描
- 边界测试

#### 第二轮：afrexai-qa-testing-engine
- 功能完整性（对照 PRD）
- 运行验证
- 性能测试（FPS ≥60）

#### 第三轮：workflow-enhancements
- 生成测试证据包
- 录屏 + 截图 + 性能报告

### 测试证据包内容
```markdown
## 测试证据包

### 📹 操作录屏
- 时长：30 秒
- 内容：启动 → 操作 → 功能演示

### 📸 功能截图
- 6-10 张关键功能截图

### 📊 性能报告
- FPS: 58-60
- 内存：78MB

### ✅ 测试清单
- [x] 游戏能启动
- [x] 控制有响应
- [x] 无控制台报错
```

### 🚨 禁止行为
- ❌ 跳过任何一轮测试
- ❌ 测试不通过就交付
- ❌ 不生成证据包
- ❌ 让用户当测试员

## 🎨 阶段 4：美术替换系统集成

### 每个游戏必须包含

#### 功能清单
- [x] 密码保护（默认：admin123）
- [x] 资源列表（自动识别）
- [x] 上传替换（真正覆盖文件）
- [x] 实时预览（替换后立即生效）
- [x] 一键部署（Git+ 推送）
- [x] 移动端友好

#### 集成方式
```html
<!-- 在游戏页面添加 -->
<button class="editor-btn" onclick="openEditor()">🎨 美术替换</button>
<script src="js/editor.js"></script>
```

#### 标准目录结构
```
/project-name/
├── index.html
├── js/
│   ├── game.js
│   └── editor.js  ← 标准组件
├── css/
│   └── style.css
├── assets/
│   ├── gems/
│   ├── ui/
│   ├── effects/
│   └── bg/
└── VERSION.json
```

### 🚨 禁止行为
- ❌ 游戏不包含美术替换系统
- ❌ 无密码保护
- ❌ 替换后不实时预览
- ❌ 用 Canvas 绘制而不是图片（无法替换）

## 📊 阶段 5：版本管理

### 版本规则
```
V[大版本].[功能版本].[BUG 版本]
  ↑        ↑          ↑
 重大更新  功能增加    BUG 修复
```

### 升级标准
| 变更类型 | 升级位置 | 示例 |
|----------|----------|------|
| 架构变更/核心玩法 | 第一位 | V1.0.0 → V2.0.0 |
| 新功能/新系统 | 第二位 | V1.1.0 → V1.2.0 |
| BUG 修复/优化 | 第三位 | V1.1.1 → V1.1.2 |

### 必须执行
1. 更新 VERSION.json
2. 创建 Git 标签
3. 生成变更日志

### 🚨 禁止行为
- ❌ 版本号不一致
- ❌ 更新后不做版本管理
- ❌ 不写变更日志

---

## 🚀 阶段 6：部署同步

### 三致原则
**GitHub 仓库版本号 = 服务器版本号 = 在线测试链接版本号**

### 一键部署流程
```
1. 保存本地 → 2. 推送 GitHub → 3. 部署服务器 → 4. 更新测试链接
```

### 一致性检查
```markdown
| 位置 | 版本号 | 状态 |
|------|--------|------|
| 本地 VERSION.json | V1.0.1 | ✅ |
| Git 标签 | V1.0.1 | ✅ |
| GitHub 远程 | V1.0.1 | ✅ |
| 服务器 | V1.0.1 | ✅ |
| 测试链接 | V1.0.1 | ✅ |
```

### 🚨 禁止行为
- ❌ 版本号不一致
- ❌ 不同步 GitHub
- ❌ 不提供在线测试链接

## 💡 经验教训（血泪史）

### 2026-03-10 消消乐开发教训

#### ❌ 犯过的错误

1. **Canvas 绘制无法替换**
   - 问题：用 Canvas 绘制宝石，无法替换图片
   - 教训：必须用`<img>`标签，不能用 Canvas
   - 解决：改用 DOM + 图片渲染

2. **链接总打不开**
   - 问题：Python HTTP 服务会自动关闭
   - 教训：必须用 Nginx 作为永久 Web 服务器
   - 解决：配置 Nginx 系统服务

3. **替换后看不到效果**
   - 问题：替换后游戏预览不刷新
   - 教训：替换后必须自动刷新游戏
   - 解决：替换后立即刷新 iframe + 添加时间戳

4. **原图预览不是真实图片**
   - 问题：用色块代替图片
   - 教训：原图预览必须显示真实游戏资源
   - 解决：从游戏目录加载真实图片

5. **挤牙膏式开发**
   - 问题：每次都等用户提醒才补充功能
   - 教训：开发前必须检索完整需求
   - 解决：创建需求检查清单

#### ✅ 正确做法

1. **开发前**
   - 检索所有相关对话
   - 列出完整功能清单
   - 逐条确认："还缺少什么吗？"

2. **开发中**
   - 一次性实现完整
   - 主动汇报进度
   - 遇到问题立即请示

3. **交付前**
   - 自己先测试 3 遍
   - 确认链接能打开
   - 对照需求逐条验证

4. **交付时**
   - "还有什么需要补充的吗？"
   - 不要等用户发现

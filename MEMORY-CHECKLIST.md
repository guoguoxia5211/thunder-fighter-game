# 🧠 记忆验证清单

_每次会话前后必须执行的检查清单_

---

## ✅ 会话开始检查

**每次收到用户消息后，先执行：**

- [ ] 读取 SESSION-STATE.md（获取活跃上下文）
- [ ] 读取 MEMORY.md（主会话才读取）
- [ ] 读取 memory/YYYY-MM-DD.md（今天 + 昨天）
- [ ] 运行 memory_search 召回相关记忆
- [ ] 对比上次会话状态，确认无遗漏

**快速验证问题：**
1. 用户上次在做什么？
2. 有待处理动作吗？
3. 有未完成的决定吗？
4. 用户表达过什么偏好吗？

---

## ✅ 会话中检查

**收到以下信号时立即写入记忆：**

| 信号类型 | 关键词 | 动作 |
|----------|--------|------|
| 纠正 | "不对"/"应该是"/"实际上" | 写 SESSION-STATE.md + 承认错误 |
| 决定 | "用 X"/"选 Y"/"决定" | 写 SESSION-STATE.md + Git-Notes |
| 偏好 | "我喜欢"/"总是"/"从不" | 写 MEMORY.md + memory_store |
| 专有名词 | 人名/地名/公司名 | 写 SESSION-STATE.md |
| 具体值 | 数字/日期/ID/URL | 写 SESSION-STATE.md |
| 担忧 | "担心"/"怕"/"不确定" | 写 SESSION-STATE.md + 安抚 |

**WAL 协议：** 先写状态，再回复！

---

## ✅ 会话结束检查

**准备结束会话前：**

- [ ] 更新 SESSION-STATE.md 最终状态
- [ ] 追加 memory/YYYY-MM-DD.md 日志
- [ ] 重要决策存 Git-Notes（如可用）
- [ ] 创建 .bak 备份（如有重大修改）
- [ ] 检查是否有待处理动作

**结束模板：**
```markdown
## 会话结束 - YYYY-MM-DD HH:MM

### 完成的任务
- ...

### 待处理动作
- [ ] ...

### 关键决策
- ...

### 下次会话优先事项
1. ...
```

---

## ✅ 心跳检查（每 2-4 小时）

**通过 HEARTBEAT.md 触发：**

- [ ] SESSION-STATE.md 是否最新？
- [ ] MEMORY.md 是否一致？
- [ ] 今日 memory/ 日志是否完整？
- [ ] 待处理动作是否有遗漏？
- [ ] 项目状态是否同步？
- [ ] 运行备份脚本（每周）

---

## ✅ 记忆一致性验证

**当用户说"记忆不对"或"你忘了"时：**

1. **立即停止** 当前任务
2. **读取** SESSION-STATE.md
3. **读取** MEMORY.md
4. **运行** memory_search
5. **对比** 用户说的 vs 记忆中的
6. **承认** 不一致（如有）
7. **询问** 正确信息
8. **更新** 所有记忆层
9. **记录** 此次事件到 memory/

**回应模板：**
```
你说得对，我检查一下记忆...

【SESSION-STATE.md 记录】...
【MEMORY.md 记录】...
【我的理解】...

哪里不一致？我立即修正。
```

---

## ✅ 项目修改前验证（VBR 协议）

**准备修改项目文件前：**

1. **读取** 当前文件内容
2. **创建** .bak 备份
3. **确认** 修改内容（向用户）
4. **执行** 修改
5. **验证** 功能正常
6. **汇报** 结果

**确认模板：**
```
⚠️ 准备修改：[文件名]

【当前状态】...
【计划修改】...
【影响范围】...
【回滚方案】有 .bak 备份

请确认：是/否
```

---

## ✅ 每周回顾（建议周日执行）

**检查项目：**

- [ ] 检查 memory/ 日志完整性
- [ ] 提炼重要内容到 MEMORY.md
- [ ] 清理过期 SESSION-STATE.md
- [ ] 运行记忆完整性检查
- [ ] 运行备份脚本
- [ ] 检查 Git-Notes（如可用）
- [ ] 回顾本周犯过的错误
- [ ] 更新经验教训

---

## 📊 记忆健康指标

| 指标 | 健康 | 警告 | 危险 |
|------|------|------|------|
| SESSION-STATE.md 更新频率 | 每次会话 | 偶尔遗漏 | 经常不更新 |
| memory/ 日志完整性 | 每天完整 | 偶有缺失 | 经常缺失 |
| 记忆一致性 | 100% | 偶有不一致 | 经常不一致 |
| 用户纠正次数 | 0 | 1-2 次/周 | >3 次/周 |
| WAL 协议遵守率 | 100% | >90% | <90% |

**目标：** 全部保持"健康"状态

---

## 🚨 紧急恢复流程

**当发现严重记忆丢失时：**

1. 立即停止所有任务
2. 向用户承认问题
3. 读取所有记忆文件
4. 请用户帮助重建上下文
5. 详细记录此次事件
6. 分析原因并防止再犯

**事件记录模板：**
```markdown
## 记忆丢失事件 - YYYY-MM-DD

### 症状
用户发现...

### 原因分析
...

### 影响范围
- 丢失的内容：...
- 受影响的项目：...

### 恢复措施
1. ...

### 预防措施
1. ...
```

*此清单必须严格遵守，违反会导致记忆丢失*

## 🔄 完整流程 (11 步 + 4 增强)

| 步骤 | 环节 | 谁来做 | 说明 |
|------|------|--------|------|
| 0 | **PRD 创建** | 我 | 列出功能清单 + 验收标准 |
| - | **开发前检查** | 我 | 确认理解一致 |
| 1 | 需求分析 | 我 | 基于 PRD |
| 2 | 执行开发 | 我 | 对照 PRD 功能清单 |
| - | **中期汇报** | 我 | 进度 50% 时，你确认方向 |
| 3 | **QA 测试** | 我 | 功能完整性 + 运行验证 + BUG 扫描 |
| - | **测试证据包** | 我 | 录屏 + 截图 + 性能报告 |
| 4 | QA 通过？ | - | 不通过→返回修复 |
| 5 | 版本管理 | 我 | 升级版本号 (V1.1.1) |
| - | **版本日志** | 我 | 写清楚这版改了什么 |
| 6 | GitHub 同步 | 我 | 推送代码 + 标签 |
| 7 | 在线测试链接 | 我 | 生成可访问链接 |
| 8 | 一致性检查 | 我 | GitHub=服务器=测试链接 |
| 9 | 汇报完成 | 我 | 附完整报告 + 证据包 |

**你需要做的：**
1. 确认 PRD
2. 确认开发前检查
3. 确认中期汇报方向
4. 看报告 + 证据包
5. 点开链接玩（可选）

## 📋 核心原则

### 1. 三致原则
**GitHub = 服务器 = 测试链接**  
版本号必须完全一致，不允许偏差。

### 2. 功能完整性
- 游戏必须可正常运行才交付
- 不擅自简化/删减任何功能
- 需求变更必须先请示

### 3. QA 测试
- 所有功能必须通过 QA 测试
- 测试不通过 → 返回修复
- 测试报告必须完整记录

### 4. PRD 原则
- 需求不明确不开工
- PRD 文档必须用户确认
- 开发对照 PRD 功能清单

### 5. 工作流增强
- 每次交付附带测试证据包
- 开工前做开发前检查
- 进度 50% 时主动汇报
- 每个版本写变更日志

## 📝 模板速查

### PRD 需求文档模板

```markdown
# 游戏需求文档 (PRD)

**项目：** [游戏名称]
**版本：** V1.0.0

## 功能清单
| ID | 功能 | 优先级 | 验收标准 |
|----|------|--------|----------|
| F01 | 玩家控制 | P0 | 响应<100ms |
| F02 | 发射子弹 | P0 | 冷却 0.3 秒 |

## 用户确认
- [ ] 功能清单正确
- [ ] 可以开始开发
```

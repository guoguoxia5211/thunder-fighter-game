## 🎯 核心原则

**记忆丢失 = 项目失败**

绝不允许再次发生因记忆丢失导致的：
- ❌ 突然不理解对话上下文
- ❌ 乱改项目内容
- ❌ 重复犯同样的错误
- ❌ 忘记关键决策

## 📋 各层详细说明

### Layer 1: SESSION-STATE.md (HOT RAM)

**位置：** `/root/.openclaw/workspace/SESSION-STATE.md`

**作用：** 活跃工作记忆，每次会话必读

**内容：**
- 当前任务
- 待处理动作
- 最近决策
- 关键上下文

**更新规则：**
| 触发 | 动作 |
|------|------|
| 用户纠正 | 立即写入纠正内容 |
| 用户决定 | 立即写入决定 |
| 用户偏好 | 立即写入偏好 |
| 任务开始 | 更新当前任务 |
| 任务完成 | 标记完成 + 写结果 |

**WAL 协议：** 先写 SESSION-STATE.md，再回复用户

---

### Layer 2: MEMORY.md (长期记忆)

**位置：** `/root/.openclaw/workspace/MEMORY.md`

**作用：**  curated 长期记忆，保持 <5KB

**内容：**
- 用户信息（姓名/偏好/时区）
- 核心目标
- 活跃项目状态
- 关键决策记录
- 经验教训
- 已安装技能

**更新规则：**
- 每周回顾整理
- 重大决策立即添加
- 保持精简，详细链接到 memory/

---

### Layer 3: memory/YYYY-MM-DD.md (每日日志)

**位置：** `/root/.openclaw/workspace/memory/YYYY-MM-DD.md`

**作用：** 原始对话日志，详细记录

**内容：**
- 当天所有重要对话
- 任务进展
- 问题与解决
- 临时决策

**更新规则：**
- 每天一个文件
- 会话结束时追加
- 保留所有细节

---

### Layer 4: Git-Notes (知识图谱)

**位置：** Git-Notes 系统

**作用：** 结构化决策和教训，可检索

**内容：**
- 技术决策（为什么选 X 不选 Y）
- 项目教训（犯过的错误）
- 用户偏好（明确表达的喜好）
- 关键上下文（人名/日期/URL）

**命令：**
```bash
# 存储决策
python3 memory.py -p . remember '{"type":"decision","content":"..."}' -t tech -i h

# 检索
python3 memory.py -p . get "关键词"
```

---

### Layer 5: .bak 自动备份

**位置：** 各关键文件的 `.bak` 副本

**作用：** 防止误删/误改

**规则：**
- 修改 MEMORY.md 前先备份
- 修改 SESSION-STATE.md 前先备份
- 保留最近 3 个版本

## 🔄 保护协议

### 1. WAL 协议 (Write-Ahead Log)

**规则：** 先写状态，再回复

```
用户输入 → 解析意图 → 写 SESSION-STATE.md → 回复用户
```

**触发信号：**
- "不对"/"应该是"/"实际上" → 写纠正
- "用 X"/"选 Y"/"决定" → 写决定
- "我喜欢"/"总是"/"从不" → 写偏好
- 人名/地名/公司名 → 写专有名词
- 数字/日期/ID/URL → 写具体值

---

### 2. VBR 协议 (Verify-Before-Reply)

**规则：** 修改项目前先验证 + 备份

```
准备修改 → 读取当前状态 → 备份 → 确认 → 执行 → 验证 → 汇报
```

**步骤：**
1. 读取相关文件
2. 创建 .bak 备份
3. 向用户确认修改内容
4. 执行修改
5. 验证功能正常
6. 汇报结果

---

### 3. 心跳检查 (Heartbeat Check)

**频率：** 每 2-4 小时

**检查项：**
- [ ] SESSION-STATE.md 是否最新
- [ ] MEMORY.md 是否一致
- [ ] 今日 memory/ 日志是否完整
- [ ] 待处理动作是否有遗漏
- [ ] 项目状态是否同步

**执行：** 通过 HEARTBEAT.md 触发

---

### 4. 确认协议 (Confirmation Protocol)

**重大改动前必须确认：**
- 删除文件/功能
- 重构核心代码
- 更换技术方案
- 修改用户明确过的偏好

**确认格式：**
```
⚠️ 准备执行重大改动

【当前状态】...
【计划修改】...
【影响范围】...
【回滚方案】...

请确认：是/否
```

## 📊 记忆刷新协议

| 上下文使用率 | 动作 |
|--------------|------|
| <50% | 正常操作 |
| 50-70% | 重要交换后写关键点 |
| 70-85% | 主动刷新 — 立即写一切重要的 |
| >85% | 紧急刷新 — 下次回复前写完整摘要 |

---

## 🚨 紧急恢复流程

**当发现记忆不一致时：**

1. **立即停止** 当前任务
2. **读取** SESSION-STATE.md
3. **运行** memory_search 召回相关记忆
4. **对比** MEMORY.md 和当前理解
5. **承认** 不一致并询问用户
6. **修正** 理解并更新所有记忆层
7. **记录** 此次事件到 memory/ 日志

**用户指令：**
- "查 SESSION-STATE.md"
- "查 MEMORY.md"
- "你忘了 XXX"
- "记忆不对"

## ✅ 检查清单

### 每次会话开始
- [ ] 读取 SESSION-STATE.md
- [ ] 读取 MEMORY.md（主会话）
- [ ] 读取 memory/YYYY-MM-DD.md（今天 + 昨天）
- [ ] 运行 memory_search 召回相关记忆

### 每次会话结束
- [ ] 更新 SESSION-STATE.md 最终状态
- [ ] 追加 memory/YYYY-MM-DD.md 日志
- [ ] 重要决策存 Git-Notes
- [ ] 创建 .bak 备份（如有重大修改）

### 每周回顾
- [ ] 检查 memory/ 日志完整性
- [ ] 提炼重要内容到 MEMORY.md
- [ ] 清理过期 SESSION-STATE.md
- [ ] 运行记忆完整性检查

## 📝 违规后果

**违反保护协议的后果：**

| 违规 | 后果 |
|------|------|
| 未执行 WAL | 记忆丢失，项目返工 |
| 未执行 VBR | 误改项目，用户测试 |
| 未心跳检查 | 状态不同步 |
| 未确认重大改动 | 信任丧失 |

**绝不再犯！**

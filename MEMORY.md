## 📋 工作流规范 (🔥 最高优先级)

### 版本管理规则
- **格式：** V1.1.1 (大版本。功能版本.BUG 版本)
- **V1.x.x** - 重大更新 (架构/核心玩法)
- **Vx.1.x** - 功能增加 (新系统/新功能)
- **Vx.x.1** - BUG 修复 (优化/修复)
- **强制：** 每次执行完更新后必须做版本管理

### 三致原则
- GitHub 仓库版本号
- 服务器版本号
- 在线测试链接版本号
- **必须完全一致，不允许偏差**

### 功能完整性原则
- ✅ **游戏必须可正常运行才交付**
- ✅ **不擅自简化/删减任何功能**
- ✅ **需求变更必须先请示用户**
- ✅ **功能完整性逐项对照检查**
- ✅ **强制运行验证 + 截图证据**

### QA 测试原则
- ✅ **所有功能必须通过 QA 测试**
- ✅ **测试不通过 → 返回修复**
- ✅ **测试报告必须完整记录**

### PRD 原则
- ✅ **需求不明确不开工**
- ✅ **PRD 文档必须用户确认**
- ✅ **开发对照 PRD 功能清单**
- ✅ **验收对照 PRD 验收标准**

### 工作流增强原则 (🔥 新增)
- ✅ **每次交付附带测试证据包**
- ✅ **开工前做开发前检查**
- ✅ **进度 50% 时主动汇报**
- ✅ **每个版本写变更日志**

### 标准流程 (11 步 + 4 增强)
0. **PRD 创建** → 【开发前检查】→ 1. 需求分析 → 2. 执行开发 → 【中期汇报】→ 3. **QA 测试** → 【测试证据包】→ 4. 版本管理 → 【版本日志】→ 5. GitHub 同步 → 6. 在线链接 → 7. 一致性检查 → 8. 汇报

## 🧠 认知系统配置

### WAL 协议 (Write-Ahead Log)
**规则：** 在回复前先写状态，再回复

| 触发信号 | 关键词 | 动作 |
|----------|--------|------|
| 纠正 | "不对"/"应该是"/"实际上" | 写 SESSION-STATE.md |
| 决定 | "用 X"/"选 Y"/"决定" | 写 + Git-Notes |
| 偏好 | "我喜欢"/"总是"/"从不" | 写 + memory_store |
| 专有名词 | 人名/地名/公司名 | 写 SESSION-STATE.md |
| 具体值 | 数字/日期/ID/URL | 写 SESSION-STATE.md |

### 复杂度评估
- ≤5 分：快速响应（无标记）
- 6-7 分：标准推理 🧠
- ≥8 分：深度推理 🧠🔥
